## Restore workflow

The client menu now contains both:

- `Restore from Hub`
- `Make Space`

`HubRestoreManager` uses the `restore_done_v2` preference and returns a
`RestoreSummary`. The one-time restore is only marked complete when the hub
returns a valid file list and the restore pass has zero failures. An empty or
temporarily unavailable hub file list remains pending and retries after a later
hub connection.

At the end of verification, the client logged:

```text
HubRestore: hub returned no files; will retry next hub connect
```

This is intentional pending behavior, not a completed restore.

## Memory issue and fix

The client crash was an Android heap OOM, not USB storage exhaustion:

- File: `20260419_131315.jpg`
- Size: 306,287,760 bytes
- Approximate client heap limit: 268 MB
- Failure: `LocalImageProcessor.processUnfixed()` called `readBytes()`

Fixes:

- `LocalImageProcessor` skips files larger than 32 MB in its full-buffer local
  metadata repair pass and catches allocation failures.
- `UsbStorageManager.thumbnailForFile()` no longer loads a complete image into
  memory. It reopens streams for EXIF, bounds decoding, and sampled decoding.

The hub subsequently synced with roughly a 10-15 MB managed heap footprint
instead of attempting a source-sized allocation.

## Preventing restore/upload loops

Restored originals can later be compressed or posterized on the phone. Those
generated derivatives must never be treated as new files to back up again.

`MediaStoreHelper.getMediaSince()` now excludes:

- Compressed replacement MediaStore IDs
- Compressed replacement display names
- Image compression names
- Video compression names
- Video poster names

Additional existing protections remain in place:

- Image and video replacements preserve the original display name.
- Poster JPEG names are tracked by `VideoSpaceManager`.
- The hub rejects poster images whose stem matches a video already on USB.
- The hub checks exact, stripped suffix, and stem-equivalent USB names.
